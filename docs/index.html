Why Batching Beats Kernel Optimizations in Microsecond AI Inference

TL;DR
When AI inference becomes deterministic (~20µs), Linux host overhead becomes visible. The biggest performance lever is not kernel optimization—it is batching.

Once inference becomes deterministic, the Linux control plane—not the model—dominates latency.

1. The Shift: From Compute-Bound to Control-Plane Bound

Traditional performance thinking assumes compute dominates. But with SRAM-style inference:

  • Execution becomes fast (~20µs)
  • Variance collapses
  • Latency becomes predictable

This pushes systems into a new regime:

Total Latency = Compute + Host Overhead

When compute stabilizes, the host becomes the bottleneck.

2. Initial Hypothesis: The Completion Path

The obvious suspect:

IRQ → softirq → wakeup → scheduler → userspace

This suggests optimizing:

  • Interrupt handling
  • Wakeups
  • Polling (CQ polling)

But measurement showed:

  • Completion latency is sub-microsecond
  • Not a dominant contributor
  • CQ polling is NOT yet justified

Conclusion: Completion is not the primary bottleneck.

3. The Real Bottleneck: Submission

Breaking the pipeline:

userspace → submit → issue → device → complete → userspace

Measured result:

  • submit → issue ≈ 5–7µs p99
  • Compute ≈ 20µs

This means submission overhead is ~25–30% of total latency.

4. A Simple Cost Model

Cost(N) = Fixed + (N × Per-request)
Cost_per_request = (Fixed / N) + Per-request

Where:

  • Fixed: syscall, ring setup, scheduling
  • Per-request: minimal work per IO
If cost is mostly fixed, the only solution is amortization.

6. Key Result

Batch=1   → ~470ns per request
Batch=16  → ~87ns per request

That’s a 7× reduction in submission overhead.

Critically:

  • p99 latency remains stable
  • Tail latency does not explode

7. Why Batching Works

Batching amortizes:

  • Syscall overhead (io_uring_enter)
  • Ring processing
  • Cache synchronization
  • Scheduler interaction

Important nuance:

The kernel still processes each request individually, but fixed costs are paid once per batch.

8. The Optimal Region

Batching has diminishing returns:

  • Too small → high overhead
  • Too large → increased latency

Observed optimum:

Batch size ≈ 8–16

9. System-Level Implication

Wrong:
"Optimize the kernel"

Right:
"Amortize system overhead"

This leads to a broader insight:

System design dominates kernel micro-optimization in the microsecond regime.

10. What This Means for AI Systems

  • Batch aggressively but carefully
  • Target latency-efficient batching window
  • Optimize architecture before kernel

Kernel work still matters—but only after batching is exploited.

11. Future Work

  • Adaptive batching
  • Native Linux validation
  • Submission-path kernel tuning

Final Takeaway

SRAM-style inference doesn’t remove latency—it reveals it. And batching is the most powerful lever we have.
